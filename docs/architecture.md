# FamilyMemoryOS 架构设计（初稿）

## 1. 文档目的

本文档用于描述 FamilyMemoryOS 的整体架构设计，明确系统的核心层次、模块职责、数据流向与后续扩展方向。

本文档不追求一次性定死所有实现细节，而是用于统一项目当前阶段的工程理解，避免后续开发中出现定位漂移、模块混乱或职责重叠。

---

## 2. 总体目标

FamilyMemoryOS 是一个本地优先、隐私优先的家族数字记忆框架。

它希望在离线或弱联网环境下，帮助家庭完成以下事情：

- 组织家族人物与亲缘关系
- 记录人物事件与家族时间线
- 保存和整理家书、聊天记录、日记、录音转写等记忆资料
- 基于现有资料进行问答与检索
- 在证据约束下生成特定人物视角的数字纪念交互
- 长期维护一个可扩展、可追溯、可人工修正的家族记忆系统

---

## 3. 核心思想

FamilyMemoryOS 不将“某个人”简单视为一个聊天角色，也不将“技能（skills）”视为最终目标。

本项目的基本判断是：

> 一个人的数字化重建，不应只有“像不像他说话”，而应同时包含：
> - 关系位置
> - 生平事件
> - 原始资料
> - 记忆片段
> - 人格画像
> - 行为模式（skills）
> - 交互边界

因此，FamilyMemoryOS 采用如下核心结构：

**Memory Layer + Graph Layer + Persona Layer + Skill Layer + Interaction Layer**

其中：
- `Memory` 负责“他留下了什么”
- `Graph` 负责“他是谁，与谁有什么关系”
- `Persona` 负责“他大致是怎样的人”
- `Skills` 负责“他通常怎样表达、判断、回应”
- `Interaction` 负责“系统最终怎样与用户交互”

---

## 4. 五层架构概览

### 4.1 Memory Layer（记忆层）
负责保存和组织原始资料及其切分后的可检索内容。

这是系统最底层、最基础的一层。

#### 主要职责
- 保存原始资料
- 保存导入后的元数据
- 保存切分后的记忆片段
- 支持来源追踪
- 为检索与问答提供基础语料

#### 典型对象
- Artifact
- MemorySegment

#### 典型输入
- 家书
- 日记
- 聊天记录
- 录音转写
- 照片说明
- 口述整理文本

#### 典型输出
- 可检索的记忆片段
- 原始资料引用路径
- 来源等级信息

---

### 4.2 Graph Layer（图谱层）
负责表示家族中的人物、关系、事件、地点与时间线。

这一层的目标不是做复杂图数据库炫技，而是提供一个稳定的“关系骨架”。

#### 主要职责
- 管理家族成员
- 管理人物关系
- 管理人物事件与家庭事件
- 支持后续时间线与关系视图
- 为人格与对话提供上下文约束

#### 典型对象
- Person
- Relationship
- Event
- Place（后续可选）

#### 典型问题
- 谁是谁的父亲/母亲/配偶/子女？
- 某位人物经历过哪些重要事件？
- 某个时期家庭发生过什么？

---

### 4.3 Persona Layer（人格层）
负责从资料与记忆中抽取并维护某位人物的相对稳定画像。

这一层不是“无限拟态”，而是“受证据支持的概括性画像”。

#### 主要职责
- 抽取语言风格
- 抽取价值偏好
- 抽取常见主题
- 抽取决策倾向
- 记录禁忌与不确定性

#### 典型对象
- PersonaProfile

#### 典型内容
- 说话直接还是委婉
- 常见表达方式
- 对稳定、风险、责任、亲情等议题的倾向
- 反复出现的人生主题
- 对某些问题的保守边界

#### 注意事项
- Persona 是“概括画像”，不是“全知还原”
- Persona 可以版本化
- Persona 应允许人工校正

---

### 4.4 Skill Layer（技能层）
负责将人物的典型行为模式提取为可复用、可组合、可约束的模块。

这是当前许多“把人蒸馏成 skill”思路，与本项目最直接对接的位置。

#### 核心理解
在 FamilyMemoryOS 中，skill 不是一个人的全部，而是某个人在特定情境下相对稳定的行为模式表达。

也就是说：

> 人 ≠ skill  
> 但 person 可以拥有一组与其记忆、人格和关系相关的 skills

#### 主要职责
- 提取表达 skill
- 提取判断 skill
- 提取情境 skill
- 在交互时为系统提供可调用行为模板

#### 推荐的 skill 分类

##### 1. Expression Skills（表达技能）
描述“怎么说”。

例如：
- 安慰人时倾向先共情还是先讲道理
- 更喜欢短句还是长句
- 是否常用某些口头表达
- 是否习惯先结论后解释

##### 2. Judgment Skills（判断技能）
描述“怎么判断”。

例如：
- 在家庭冲突中优先维护和气
- 面对职业选择偏向稳定
- 面对风险时倾向谨慎
- 面对晚辈问题时强调长期积累

##### 3. Situational Skills（情境技能）
描述“在什么身份和场景下怎么回应”。

例如：
- 作为父亲如何回应孩子的困惑
- 作为长辈如何劝学
- 作为配偶如何表达关心
- 在纪念日场景下如何表达情感

#### skill 的来源
- 原始文本中的稳定模式
- 家人访谈中的总结
- 人工整理的规则
- 由 persona 推导出的高置信模式

#### skill 的形式
后续工程实现上，skill 可以采用结构化文件或数据库对象表示，例如：
- `skills/*.md`
- `skills/*.yaml`
- 数据库中的 `Skill` 表

---

### 4.5 Interaction Layer（交互层）
负责最终面对用户时的交互逻辑。

它不直接生产知识，而是调度 Memory、Graph、Persona、Skills，并决定输出策略。

#### 主要职责
- 接收用户问题
- 判断当前交互模式
- 检索相关记忆片段
- 组合人物画像与 skills
- 生成最终回答
- 展示证据和来源等级

#### 交互模式
FamilyMemoryOS 计划支持三类模式：

##### 1. Strict Evidence Mode
- 只基于资料回答
- 尽量不拟态
- 更像档案问答系统

##### 2. Evidence-Constrained Persona Mode
- 允许使用特定人物视角和语气
- 但必须受到资料与 persona 约束
- 对未知内容要保守

##### 3. Memorial Companion Mode
- 允许更柔和的纪念性生成
- 但必须明确提示这不是事实保证或意识延续

## 5. 数据流概览

### 5.1 导入阶段
输入：
- 本地文本资料
- 手工整理记录
- 结构化人物与事件信息

流程：
1. 导入原始资料
2. 生成 Artifact
3. 进行文本切块
4. 生成 MemorySegment
5. 标注所属人物 / 时间 / 来源 / 隐私级别
6. 供检索、persona 抽取与 skill 抽取使用

---

### 5.2 建模阶段
输入：
- MemorySegment
- Person / Relationship / Event
- 人工校正信息

流程：
1. 构建家族关系图谱
2. 构建人物时间线
3. 生成或更新 PersonaProfile
4. 提取或维护 Skill 集合

输出：
- 可解释的人物数字画像
- 可复用的 skill 模块
- 可检索的关系上下文

---

### 5.3 交互阶段
输入：
- 用户问题
- 目标人物
- 交互模式

流程：
1. Interaction 层识别问题和模式
2. 调用 Graph 获取关系与上下文
3. 调用 Memory 检索相关片段
4. 调用 Persona 提供风格与价值约束
5. 调用 Skills 提供典型行为模式
6. 组合生成回答
7. 返回答案 + 依据片段 + 来源等级

---

## 6. 为什么 skill 只是系统的一部分

在 FamilyMemoryOS 中，skill 不应被误认为是整个数字人格。

原因包括：

### 6.1 skill 只能表示“可复用模式”
它擅长表示：
- 常见说法
- 常见判断
- 常见应对方式

但不擅长完整表示：
- 生平经历
- 代际关系
- 时期变化
- 内在矛盾
- 未留下资料的复杂经验

### 6.2 没有 Memory 和 Graph，skill 会失根
如果只有 skill，而没有：
- 原始资料
- 时间线
- 家族关系
- 来源等级

那么系统就会滑向单纯的角色扮演，而不是家族记忆系统。

### 6.3 没有 Persona，skill 会碎片化
没有人格层，skill 往往只是一堆局部规则，很难形成稳定一致的回答风格。

因此：
- `Memory` 提供依据
- `Graph` 提供关系骨架
- `Persona` 提供人物稳定性
- `Skills` 提供情境行为模式
- `Interaction` 提供最终编排

## 7. 当前阶段的工程优先级

为了避免项目一开始过度复杂，当前阶段只优先实现以下结构：

### 第一优先级
- Graph Layer 基础对象
- Memory Layer 基础对象
- Interaction Layer 的最小问答能力

### 第二优先级
- PersonaProfile 的基础结构
- 基础来源等级展示

### 第三优先级
- 简单 skill 结构
- skill 与 persona 的初步结合

### 暂缓内容
- 复杂自动抽取
- 多模态深度解析
- 高度拟真的人格生成
- 高复杂度权限策略
- 分布式部署

## 8. 初版工程映射建议

建议当前代码结构大致映射如下：

```text

├─ api/           # HTTP 接口
├─ core/          # 配置、基础组件

├─ schemas/       # Pydantic schema
├─ services/      # 业务逻辑

├─ imports/       # 导入与切块

├─ persona/       # PersonaProfile 相关
├─ skills/        # 技能抽取与表示（后续）
